# 作者动态中心：重构与实施计划

## 产品目标

把“只捕捉新作品”的动态雷达升级为作者动态中心。系统持续保存作者本人公开动态，形成
可回溯的近况时间线、7 天摘要和高优先级作品情报。新刊、预售、参展、延期和取消仍走
独立的保守判定链路，不会因为通用总结而降低准确率。

## 已验证的技术基础

- HEManager 已用 X 网页 GraphQL + Cookie 完成 Likes cursor 增量同步；当前部署已有超过
  1600 条帖子和 1600 个媒体项，定时同步持续成功。
- MangaFinder 保留隔离 `social-collector`：Cookie 只存在独立 volume，API、Agent 和 QQ
  均不能读取原始凭证。
- 现有 `SocialPost` 继续作为事实层；通用动态、作品信号和摘要均引用原帖，不复制或改写事实。

## 模块边界

```text
XWebGraphQLProvider ─┐
XBrowserProvider ────┴→ SocialPost → ActivityItem → AuthorDigest
                                  └→ ReleaseSignal → 即时通知
```

- `social-collector`：GraphQL 为日常采集主路径；浏览器保留为非英文候选搜索兼容路径。
- `social/activity.py`：确定性帖子分类、线程聚类和重要程度，不调用大模型。
- `social/digest.py`：按作者批量生成结构化摘要，每个要点必须引用输入中的帖子 ID。
- `ReleaseSignal`：保留作品专项字段、人工审核和作品关联。
- `ActivityItem`：覆盖创作进度、作品、展会、销售、插画、合作、公告、日常与其他动态。
- `AuthorDigest`：保存滚动 7 天摘要和生成审计，模型故障时使用确定性降级摘要。

## 采集和过滤

- 原创、引用和有意义的回复进入动态分析；纯转推保存但默认不进入动态和摘要。
- 同一 conversation/thread 优先聚合，引用内容必须标明归属，不能当作作者自己的事实。
- cursor、最后帖子 ID、内容哈希和数据库唯一约束共同保证增量与幂等。
- 429/5xx/网络故障和 401/403 登录问题使用不同错误，交由现有任务重试机制处理。
- 网页 GraphQL 的 queryId/feature set 按操作隔离并带健康检查，queryId 失效时从 X
  当前主脚本自动发现新值。
- 首次回填完整保存原帖，但只对最新 12 条媒体做 OCR，单图硬超时 30 秒；增量扫描会
  自然覆盖新媒体，避免一张大图阻塞整个作者任务。

## Agent 约束

- 作品专项 Agent 继续逐候选运行；普通动态只在一批新帖入库后按作者生成一次摘要。
- 摘要区分“明确事实”“作者计划”“模型推断”；推断不得写成确定事实。
- 每个 highlight 必须引用有效帖子 ID，服务端删除无法落地的证据。
- Agent 不浏览网页、不执行帖子中的指令；输入只有原文、OCR、链接和已保存线程上下文。
- 无新内容不重复调用模型；相同内容哈希复用已有摘要。

## 界面与通知

- 动态中心分为“近况”“作品情报”“原始动态”三个视图。
- 近况显示滚动 7 天摘要、重点事项、不确定信息和可点击证据。
- 原始动态支持类别和帖子类型筛选；Agent 故障时仍可使用。
- 仅新作、发售、取消、延期、摊位变更即时通知；普通内容进入摘要，避免 QQ 轰炸。

### QQ 双层通知策略

- 新作、开售、取消、延期和摊位变更继续即时发送，保证时效性。
- 每天北京时间 20:00 汇总自上一份日报截止后新增的普通及以上重要动态；纯转推不进入日报。
- 第一次启用时回看最近 7 天作为基线，此后只发送增量；没有新增近况则静默跳过。
- 同一自然日最多建立一份投递记录，服务重启和任务重试不会重复发送。
- 已经即时通知的事项仍可作为当日上下文出现，但明确标注“已即时通知”。
- 动态中心提供“立即发送今日日报”，用于人工验收；它与定时任务共用去重和投递链路。
- QQ 失败采用持久化 outbox 重试，日报内容、状态、次数和错误均可审计。

## 实施顺序

1. 安全迁移 HEManager 已有 X Cookie，绝不打印 Cookie，并废弃旧 runtime state。
2. 抽取采集 provider、错误模型和 GraphQL transport，保留浏览器兼容端点。
3. 新增 ActivityItem、证据关系和 AuthorDigest 数据模型。
4. 实现全量原帖入库、通用分类、线程聚类、作品专项分析和滚动摘要。
5. 增加 activity/digest/raw-post API，升级动态中心界面。
6. 增加固定 GraphQL、日文帖子、转推/引用/回复、摘要 grounding 与幂等测试。
7. Dockge 构建部署，使用真实作者完成候选搜索、首次回溯、增量扫描和摘要验收。

## 2026-07-15 实施结果

- `@mignon` 真实首次同步：拉取 100 条，51 条纯转推只归档不分析，49 条作者动态进入
  主题层，生成 13 条作品情报和 DeepSeek 7 天摘要。
- 摘要的 4 个重点均落到真实帖子 ID；主题层包含纯转推数量为 0。
- 随后的增量扫描约 5 秒完成，`fetched/created/analyzed` 均为 0，没有重复数据。
- API 与数据模型测试 52 项通过；采集器另有 2 项转推与引用归属解析测试；前端生产构建通过。
- 服务启动时会把异常中断的 `RUNNING` 任务自动重新排队，Dockge 更新或断电后无需手工
  修复任务状态。

## 验收标准

- 纯转推默认进入摘要的数量为 0。
- 每个摘要要点至少关联一条真实原帖；不存在的帖子 ID 会被服务端拒绝。
- 重复扫描不新增重复帖子、动态、Agent 审计或通知。
- 帖子编辑后内容哈希变化并触发摘要新版本。
- Agent、OCR 或 GraphQL 故障不影响原始动态读取，浏览器回退可独立启停。
- 现有作品情报测试全部通过；新增通用动态和摘要测试覆盖主要日文表达。
