# MangaFinder 作品聚合与多版本设计计划

## 1. 目标

把抓取到的记录拆成三个稳定层级：

1. **作品（Work Group）**：主页只显示一次，例如《美丽新世界》。
2. **版本（Edition / Variant）**：保留不同汉化组、语言、上色、无码、修正版、DL 版、上传批次等差异。
3. **来源（Source）**：版本在 WNACG、MangaDex 等网站上的具体页面，是章节查询和下载的最终定位依据。

聚合不会删除原始抓取记录，也不会把下载指向模糊的“作品”；下载始终选择明确的版本和来源。

## 2. 设计原则

- **可逆**：自动合并错误时可以拆分，人工处理不会被下一次抓取覆盖。
- **宁可少合并，不可误合并**：系列续作、总集篇、重制版不能仅因标题相似就强制合并。
- **保留证据**：保存标准化标题、版本标签、匹配分数和匹配原因。
- **来源无关**：匹配引擎位于聚合层，新增网站不需要重写主页逻辑。
- **渐进增强**：标题规则立即可用；封面感知哈希作为第二证据，不依赖单一图片 URL。
- **原始数据不丢失**：原始标题、封面、元数据和来源页面继续保存在版本记录中。

这一结构参考了媒体库对“同一内容的多个版本”的处理方式：Jellyfin 将版本聚合但仍保留版本选择和人工分组；Komga 用哈希识别重复项并保留人工管理入口。

## 3. 数据模型

### `work_groups`

主页作品实体，保存代表标题、代表封面、描述、年份、状态、语言、标签以及更新时间。

### `work_group_members`

把现有 `works`（抓取版本）关联到作品实体，记录：

- `confidence`：自动匹配置信度；
- `match_method`：`new`、`title_exact`、`title_fuzzy_cover`、`manual`、`backfill`；
- `is_manual`：人工关系受保护，自动重建不得覆盖。

使用独立关联表而非给旧表加必填列，现有 SQLite 数据可以无损、无停机地迁移。

### `work_fingerprints`

每个抓取版本的匹配特征：

- 去除作者、站点上传说明和版本词后的标准标题；
- MangaDex alternate titles 等标题别名；
- 从原始标题提取的版本标签；
- 兼容旧数据的封面 dHash（可空）；
- V2 多视角封面指纹：图片尺寸、全图与常见比例裁切的 dHash、pHash 和颜色直方图；
- 特征更新时间。

### `merge_suggestions`

中等置信度匹配不自动合并，生成待审核候选。支持接受和拒绝；拒绝结果会保留，避免每次刷新重复建议。

## 4. 匹配管线

### 4.1 标题标准化

- Unicode NFKC、大小写统一、繁简不作为强制等价；
- 将 `×/✕/✖` 统一为 `x`，并移除作者/社团前缀、展会编号、章节数、完结状态；
- 提取而不是丢弃版本信息：汉化组、翻译、上色/全彩、无码/无修正、AI 修正、DL/扫图、语言、修订版和不完整版；
- 保留可能改变作品身份的词：续、第二部、总集篇、番外、重制版等。

### 4.2 作品编号硬门禁

- 从清洗后的标题提取作品编号签名，`#01` 与 `#1` 统一为 `(1)`；
- 双语标题中重复出现的同一编号只记录一次，例如 `#02 | 小腹 #02` 仍为 `(2)`；
- `C106`、`AC2` 等展会编号、四位年份、`1080p/4K` 和 `V3` 修订号不进入作品编号签名；
- 编号签名不同，或一方有编号而另一方没有编号时，禁止自动合并，标题和封面分数都不能越过该门禁；
- 人工合并关系优先于自动规则，刷新不会推翻用户明确做出的纠正。

### 4.3 候选范围

优先只比较同一订阅作者下的作品，避免全库同名作品互相污染。MangaDex 的 alternate titles 参与跨语言匹配。

### 4.4 分层证据与自动合并阈值

- 编号门禁通过后，标准标题或任一别名完全一致：高置信度自动合并；
- 标题模糊匹配阈值为 `0.94`，短标题不能仅靠模糊分数合并；
- 标题相似并且 V2 封面视觉距离不超过 `10`：组合证据自动合并；
- 日文/罗马字等跨语言标题允许用强封面证据合并，但要求同一作者、编号签名一致、年份不冲突、页数不冲突，V2 视觉距离不超过 `8`；
- 横图、竖图和站点缩略图会按 `3:4`、`1:1`、`4:3`、`16:9` 生成有限数量的裁切指纹，匹配时组合 dHash、pHash 与颜色分布；
- 旧版单一 dHash 仍可提供正向证据，但不再单独产生“封面明显不同”的负向结论；
- V2 距离 `0–8` 为强相似、`9–13` 为弱相似、`14–22` 为不确定，只有可靠比较达到 `23` 才作为封面不同的软冲突；
- 指纹只在版本缺失或封面 URL 变化时重新下载；升级后通过限并发后台任务渐进回填，避免阻塞 API；
- 已知占位图、默认封面和无封面图片不参与感知哈希，避免整个来源被同一占位图串联；
- 只有短标题相同、年份冲突、编号冲突或出现续作标识：不自动合并；
- 中等相似度：写入 `merge_suggestions` 等待人工确认；
- 低相似度：创建新作品。

封面不能绕过编号门禁；无编号作品使用强封面证据时仍受同作者、年份和页数约束。

## 5. API 与界面

### 主页

- `GET /api/work-groups` 返回聚合作品，不再为每个上传版本重复显示卡片；
- 卡片展示去重后的来源标签、版本数量、最新来源日期；
- 代表封面和标题按信息完整度选择，后续可人工固定。

### 作品详情

- `GET /api/work-groups/{id}` 返回所有版本和来源；
- 按来源筛选：全部、WNACG、MangaDex 等；
- 每个版本展示原始标题、封面、版本标签、来源、更新时间和原站链接；
- 先选择版本，再查询该版本来源的章节和下载，不再默认使用数组第一项。

### 人工纠错

- 合并两个作品组；
- 从作品组拆出某个版本；
- 查看、接受或拒绝聚合候选；
- 人工关系标记为受保护，定时刷新不会重新拆散。

## 6. 代表元数据选择

- 标题：优先干净、信息噪声少且非空的标题；
- 封面：优先可访问且分辨率较好的封面，首版先以非空和来源优先级选择；
- 描述：优先非空且较完整的描述；
- 标签：所有版本去重合并；
- 时间：取所有来源中最新的 `source_updated_at`，主页按该时间倒序；
- 来源标签：按 provider 去重，不把同站多个上传误显示成多个网站。

## 7. 迁移与回滚

1. 先创建四张新表，不删除或改写旧表。
2. 启动时为尚未归组的旧数据执行幂等回填。
3. 旧 `/api/works` 暂时保留，下载 API 继续兼容现有任务。
4. 新前端切换到 `/api/work-groups`。
5. 若需要回滚，只需切回旧接口；旧作品和来源数据没有丢失。

## 8. 测试与验收

- 标题清洗和版本标签表驱动测试；
- 完全相同、不同汉化/上色/无码版本应合并；
- 续作、总集篇和短标题不得误合并；
- 同一网站多个 external ID 均保留为独立版本；
- 多网站来源标签正确去重；
- 来源筛选、章节查询、精确下载完整走通；
- 人工合并/拆分在再次刷新后仍保持；
- 现有数据库回填幂等，Dockge 重启不产生重复组。

## 9. 实施顺序

1. 新数据表、标题指纹与匹配引擎；
2. 发现流程接入与旧数据回填；
3. 聚合作品列表、详情和人工纠错 API；
4. 主页卡片、来源标签、详情抽屉和来源筛选；
5. 回归测试、真实数据抽查、Dockge 部署。
