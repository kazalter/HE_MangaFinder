# MangaFinder 聚合 Agent V2 优化计划

> 制定日期：2026-07-13  
> 当前阶段：Phase A-D 与候选生命周期已实现；Phase E 本地多视觉指纹需在评测样本积累后决定

## 1. 本次问题复盘

用户观察到的候选为：

- `アコちゃんサンタのプレゼント ブルーアーカイブ`
- `サンタアスナとカリンのプレゼント ブルーアーカイブ`

当前字符序列相似度为 `0.75`，主要来自共同片段：

- `サンタ`
- `のプレゼント ブルーアーカイブ`

移除作品语境 `ブルーアーカイブ` 后，相似度降为 `0.60`；再移除通用表达
`プレゼント` 后，两个核心标题的相似度仅为 `0.40`。这说明当前算法把“共同题材、共同系列、
共同普通名词”错误地当成了作品身份。

该候选的真实证据为：

| 证据 | 左侧 | 右侧 | 判断价值 |
|---|---|---|---|
| 原始标题 | アコちゃん… | アスナとカリン… | 核心角色不同 |
| 作者 | ラマンダ | ラマンダ | 仅为候选前提，不是同作证据 |
| 来源 | WNACG | WNACG | 中性，同站可有大量不同作品 |
| 页数 | 9 | 4 | 明显差异，应该成为负向证据 |
| 封面 dHash 距离 | 25 | 25 | 封面明显不同，应该成为负向证据 |
| 编号 | 均不存在 | 均不存在 | 未知，不能标记为“编号一致” |

DeepSeek 的原始结果实际上是 `different_work / 0.95`，原始理由也明确提到了核心标题和页数不同。
界面容易产生误解，是因为：

1. 只显示“置信度 95%”，没有强调这是“不同作品的置信度”；
2. 服务端用固定中文模板覆盖了模型的具体理由；
3. 候选自身仍显示规则标题相似度 75%，让人感觉系统先入为主地认为它们很像。

## 2. 已确认的根因

### 2.1 标题相似算法被共同后缀带偏

当前候选生成和 Agent 证据都使用整串 `SequenceMatcher`。它不了解以下字段的不同角色：

- 核心作品名；
- 作者、社团和活动前缀；
- 原作/系列名，例如 `ブルーアーカイブ`；
- 角色名；
- 汉化、上色、无码、语言等版本标签；
- “圣诞节、礼物、番外、短篇”等高频普通表达。

长而相同的系列后缀会压过真正不同的核心标题。

### 2.2 空编号被错误记为编号一致

当前逻辑把两侧空编号集合 `{()}` 相等视为 `number_match`。在现有 23 条成功 Agent 审核中，
有 14 条出现了这种虚假正证据。

正确规则应为：

- 两边都有非空编号且完全一致：`number_match`；
- 两边都有非空编号且不同：`number_mismatch`；
- 任意一边无编号：编号未知，不产生正证据或冲突。

### 2.3 非独立信息被当成同作证据

- 作者一致：系统本来就是按作者生成候选，应视为前置条件；
- 来源重叠：同一网站存在不同作品和重复上传，应视为中性信息；
- 语言/版本差异：只能解释“为什么元数据不同”，不能证明“为什么是同一作品”。

这些字段目前会让模型感觉正证据很多，实际独立身份信号可能只有标题一项。

### 2.4 负向视觉和页数信息表达不足

当前只有 `cover_hash_match`，没有 `cover_dissimilar`；封面距离大于 10 时，模型只看到一个没有阈值说明的数字。
页数冲突条件对短篇过宽，`9 vs 4` 因边界条件没有产生 `page_count_mismatch`。

### 2.5 输入有原始标题，但缺少原始来源元数据和语义分层

Agent 已收到：

- 原始标题、标准化标题和别名；
- 作者、标签、语言、页数、年份；
- 来源与外部编号；
- 封面哈希和最小距离。

Agent 尚未收到或未被明确区分：

- 标题核心、系列/原作语境、角色词、活动、社团；
- 来源 `raw_metadata` 中经过白名单筛选的 `artists / groups / parodies / characters / scanlator`；
- 每个版本之间的封面距离矩阵，而不只是最小值；
- 页数差值、比例和适用阈值；
- 某个词在该作者全部作品中的出现频率，用于识别通用后缀；
- 当前证据哪些是身份正证据、哪些是背景信息、哪些是负证据。

### 2.6 提示词和校验器缺少置信度约束

当前提示词没有高质量正反例，也没有说明：

- 相同系列名不等于相同作品；
- 同作者、同来源不是独立正证据；
- 只有标题相似时必须选择 `uncertain`；
- 封面明显不同需要哪些其他证据才能解释为版本差异。

服务端目前只验证字段是否合法，没有根据证据强度限制 `same_work` 的最高置信度。

### 2.7 审核解释被固定模板削弱

模型原始理由保存在审计数据中，但界面只显示服务端生成的通用模板。结果是关键细节丢失，
用户无法知道模型究竟比较了哪些核心词，也难以发现模型幻觉。

## 3. V2 设计原则

1. 候选召回和最终判定分开：候选可以稍宽，`same_work` 必须保守。
2. 作者一致、来源一致、语言差异不计作独立身份正证据。
3. 没有编号不等于编号一致；缺失值永远是未知。
4. 相同系列/原作/活动只能算背景语境，不能主导标题分数。
5. 云端 Agent 默认仍不接收成人封面图片；视觉比较在本地完成并输出可解释数值。
6. Agent 结果仍为只读建议，人工确认机制保持不变。
7. 每次证据 Schema、提示词或校准逻辑变化都提升版本号并使旧缓存失效。

## 4. 分阶段实施

### Phase A：立即修复确定性证据错误

目标文件：

- `modules/agent_review/candidates.py`
- `modules/agent_review/schemas.py`
- `modules/agent_review/grounding.py`
- `modules/catalog/aggregation.py`

工作项：

1. 修复空编号产生 `number_match` 的错误。
2. 仅在两边均有编号时才允许编号一致或冲突。
3. 移除 `provider_overlap` 的正证据身份；保留为背景元数据。
4. 把 `author_match` 标记为候选前提，不计入独立证据数量。
5. 增加负向信号：
   - `cover_dissimilar`：代表性封面距离达到明确阈值；
   - `page_count_mismatch`：同时使用绝对差值和比例，覆盖 `9 vs 4`；
   - `core_title_mismatch`：核心标题主体明显不同。
6. 不把“版本标签不同”和“语言不同”视为同作证明，只用于解释 relation。
7. 将证据 Schema 升级为 `v2`，提示词版本升级为 `v6`，旧审核缓存自动失效。

### Phase B：标题身份解析器 V2

新增独立模块，避免继续把复杂逻辑堆进 `aggregation.py`：

```text
modules/catalog/title_identity.py
├── parse_title_parts()
├── normalize_identity_core()
├── context_terms()
├── number_signature()
└── weighted_title_similarity()
```

每个标题拆成：

```json
{
  "raw": "原始标题",
  "creator": ["作者", "社团"],
  "event": ["C103"],
  "identity_core": "真正作品名",
  "context": ["原作/系列"],
  "characters": ["角色"],
  "numbers": [2],
  "variants": ["汉化", "上色"],
  "language": "zh-Hant"
}
```

标题评分改为加权组合：

- 核心标题精确/别名匹配：强信号；
- 核心标题字符 n-gram 或罗马字匹配：中强信号；
- 作品编号一致：仅在非空时为强信号；
- 系列、角色、活动、通用后缀：低权重背景信号；
- 作者作品集中高频出现的词：使用文档频率降权；
- 仅共同后缀相同、核心前部不同：产生负向信号。

候选生成不再使用单一的整串相似度阈值，而是要求：

- 核心标题达到最低相似度；或
- 存在强视觉证据并且编号不冲突；或
- 来源提供明确跨语言别名。

### Phase C：证据包 V2

Agent 输入保留原始数据，同时增加解释层，不只发送规则结论：

1. 每个版本同时提供原始标题和解析后的标题字段。
2. 白名单加入来源原始元数据：
   - `altTitles`
   - `artists`
   - `groups / circle`
   - `parodies / series`
   - `characters`
   - `scanlator`
   - `page_count`
   - 来源上传时间（明确标注不是作品发行年份）
3. 视觉证据改为：
   - 各版本封面哈希；
   - 代表性封面之间的 dHash/pHash 距离；
   - 最小、最大和中位距离；
   - 是否存在占位图或缺图。
4. 页数证据同时提供原值、最小差值和差值比例。
5. 将证据分为：
   - `identity_support`
   - `context_only`
   - `difference_support`
   - `unknown_fields`
6. `rule_confidence` 和 `rule_reasons` 不再作为 Agent 的先验结论；仅标注为
   `retrieval_reason`，提示词明确要求不得据此判断同作。

### Phase D：提示词与服务端校准

提示词 V6 增加以下明确规则和少样本：

1. 同作者、同来源、同系列不能单独证明同一作品。
2. 相同通用后缀但角色或核心标题不同，优先判为不同作品。
3. 空编号不得称为编号一致。
4. 只有一个弱信号时必须 `uncertain`。
5. 封面不同可能来自重绘/合集/汉化，但需要标题核心、编号、页数或别名等独立证据解释。
6. 输出 `identity_title_left`、`identity_title_right`、`shared_context` 和
   `decisive_differences`，让人工可直接检查模型理解。

服务端增加不可绕过的置信度校准：

- `same_work >= 0.90`：至少一个强身份信号和另一个独立支持信号；
- 只有标题模糊相似：最高只能为 `uncertain / 0.65`；
- 封面明显不同且页数明显不同：除非有精确别名或核心标题精确匹配，否则不能输出高置信度同作；
- 模型输出不满足证据门槛时，由服务端降级为 `uncertain`，并记录降级原因；
- `different_work` 的高置信度必须引用至少一项明确差异，而不是把正向证据当成反向理由。

模型原始理由不再被覆盖。界面显示：

- 模型的具体理由；
- 服务端校验后的中文证据摘要；
- 是否发生置信度降级；
- “同一作品置信度”或“不同作品置信度”的完整标签。

### Phase E：本地视觉证据增强

不向 DeepSeek 发送封面图片。服务端本地计算：

- dHash：保留当前轮廓比较；
- pHash：增强缩放和轻度编辑鲁棒性；
- 颜色直方图：辅助区分构图完全不同的封面；
- 可选的裁剪后相似度：应对汉化标题条、边框和水印；
- 多版本封面聚类：避免只取“所有配对中的最小距离”造成偶然命中。

视觉不同默认为软冲突，因为同一作品的重制版或合集可能更换封面；只有与标题核心和页数差异共同出现时，
才成为强否决依据。

### Phase F：反馈评测与候选生命周期

1. 将人工“确认合并、保持分开、拆分”转换成带来源的评测标签。
2. 不把历史自动合并直接当成真值，必须人工抽样确认。
3. 建立固定回归集，覆盖：
   - 日文与罗马字；
   - 日文与中文译名；
   - 汉化、上色、无码、缺页；
   - 同作者同系列下的不同短篇；
   - 相同通用后缀；
   - `#01 / #02`、卷号、合集；
   - 同封面不同编号；
   - 不同封面的同一版本。
4. 扫描导致证据哈希变化后，旧 Agent 结果在界面标记为“已过期”，不继续当成当前结论。
5. 只有新增或证据变化的候选才重新调用 Agent。
6. 在离线评测达标前继续保持 `AUTO_APPLY=false`。

## 5. 必须加入的回归样本

| 样本 | 期望结果 |
|---|---|
| `アコちゃんサンタのプレゼント` vs `サンタアスナとカリンのプレゼント` | 不同作品；共同系列/礼物后缀不算身份 |
| `Nure Onaka` vs `ぬぎおなか` 且封面距离 26 | 不得仅凭标题相似判高置信度同作 |
| 两边均无编号 | 不产生 `number_match` |
| `Onaka Zukushi 2` vs `おなかづくし2`，封面距离 7、页数接近 | 同一作品高置信度 |
| `JK x ONAKA #01` vs `#02` | 硬冲突，模型不调用 |
| 同核心标题、不同语言和汉化组、页数小幅差异 | 同作的不同版本 |
| 相同系列和作者，但核心角色/标题不同 | 不同作品或人工复核 |

## 6. 验收指标

第一目标是精确率，不追求一次找出全部漏聚合：

- 人工标注回归集上，高置信度 `same_work` 误合并为 0；
- 没有强身份信号时，不允许 `same_work >= 0.80`；
- 空编号误报 `number_match` 为 0；
- 共同系列后缀导致的候选数量显著下降；
- 证据变化后旧结论明确显示过期；
- Agent 界面能看到原始标题、核心标题、正向证据、负向证据和具体理由；
- 所有人工拒绝约束继续永久生效；
- 后端、前端和回归集测试全部通过。

在考虑任何自动合并前，还需满足：

- 高置信度同作精确率至少 99%；
- 至少 100 个经过人工确认的正负样本；
- 新版本先以 shadow mode 运行一轮，只记录不改变现有候选；
- 自动应用阈值继续不低于 0.98，并且要求两个独立身份信号。

## 7. 推荐实施顺序

1. Phase A：修复空编号、页数/封面负信号和伪正证据。
2. Phase B：建立独立标题身份解析器，替换整串 SequenceMatcher 候选分数。
3. Phase C：升级证据包并纳入来源白名单原数据。
4. Phase D：更新提示词、置信度校准和可解释界面。
5. Phase F：先建立评测集并 shadow 测试。
6. Phase E：根据评测结果决定是否需要更复杂的本地视觉算法。

这套顺序先解决确定性错误，再增加模型信息量，最后才引入更重的视觉能力，便于定位每次改动是否真正提高准确率。
